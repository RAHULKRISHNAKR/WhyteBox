ABSTRACT


   The WhyteBox project aims to bridge the gap between complex deep learning models and human interpretability through a modular visualization ecosystem. In its first phase, the project focuses on developing a robust backend pipeline capa- ble of extracting, converting, validating, and serving structural information from neural network models across multiple frameworks - including PyTorch, Tensor- Flow/Keras, and ONNX. The backend automatically detects model architectures, infers layer shapes and connections, and converts them into a universal JSON schema optimized for visualization. A built-in validation engine ensures struc- tural integrity and consistency across frameworks, while a REST API and CLI interface enable seamless integration with downstream visualization tools. This phase establishes the technical foundation for the next stage of the project, where 3D Visualization and Explainable AI (XAI) capabilities will be implemented to enable interactive exploration and interpretation of neural networks. By combin- ing automation, cross-framework compatibility, and scalability, WhyteBox aims to become a reliable backend solution for model introspection and explainability.

 Chapter 1 Introduction

1.1 Introduction

   Deep learning has revolutionized fields such as computer vision, natural lan- guage processing, and data analytics. However, as neural networks become deeper and more complex, understanding their internal structure and behavior becomes increasingly difficult. Most existing visualization tools either focus on simple architectural diagrams or lack cross-framework compatibility, limiting their prac- tical use for developers and researchers.
   The WhyteBox project addresses this gap by providing a modular backend system that extracts, standardizes, and serves detailed structural information about neural network models. Designed for interoperability, the system supports PyTorch, TensorFlow/Keras, and ONNX frameworks, enabling unified visualiza- tion and analysis across different model types.
   The backend forms the core infrastructure for the upcoming 3D visualization and explainability modules. It automates layer detection, connection mapping, and data conversion into a universal JSON schema, making model visualiza- tion seamless and framework-agnostic. In essence, WhyteBox transforms opaque neural architectures into interpretable, structured data ready for advanced visu- alization and explainable AI integration.



1.2 Objective

   The primary objective of the WhyteBox backend is to establish a universal, extensible foundation for deep learning model visualization and interpretability. The goals of this phase include:
� Model Extraction: Identify and extract layer-wise information (type, parameters, shapes, connections) from neural network models.
� Data Standardization: Convert model data into a universal JSON format that preserves both structural and metadata integrity.
� Validation Pipeline: Ensure consistency and correctness of the converted data through schema-based validation.
� API & CLI Integration: Enable interaction via a REST API and command-line interface, supporting both developers and automated tools.
� Visualization Readiness: Generate output optimized for downstream visualization modules, forming the basis for 3D interactive rendering in later phases.


1.3 Background

   The WhyteBox project is built upon multiple modern technologies that enable the extraction, conversion, and visualization of neural network architectures. To understand the technical foundation of this system, it is essential to review the core components and frameworks that make such interoperability possible.

1.3.1 Deep Learning Frameworks

   Deep learning frameworks such as PyTorch, TensorFlow/Keras, and ONNX provide the building blocks for creating, training, and exporting neural network models. Each framework represents models in distinct internal formats:
� PyTorch uses a dynamic computation graph and serialized .pt or .pth
models.


� Keras/TensorFlow rely on static graphs stored as .h5 or .pb files.

� ONNX (Open Neural Network Exchange) serves as an interoperability stan- dard that allows model conversion between different frameworks.

   WhyteBox's backend incorporates specialized extractors for each framework, capable of automatically identifying the model type, extracting layers, tensor shapes, parameters, and connection topologies. This ensures consistent data re- trieval irrespective of the model's origin.

1.3.2 Model Representation and JSON Schema

   To achieve framework independence, WhyteBox converts all extracted model data into a universal JSON schema. This schema defines a standardized structure for representing:

� Layers: type, parameters, shape, activation

� Connections: source-destination relationships

� Metadata: framework info, model name, layer count

   The use of a strict schema validator ensures that every model representation is syntactically and semantically correct before being passed to visualization mod- ules. This approach makes WhyteBox compatible with any frontend visualization engine and lays the groundwork for scalable 3D representations.

1.3.3 REST API and CLI Architecture

   The WhyteBox backend exposes its functionality through two primary inter- faces:

� REST API (via Flask): Enables web or frontend clients to upload mod- els, trigger conversion, and retrieve JSON output through HTTP endpoints.

� Command-Line Interface (CLI): Provides developers with a lightweight terminal tool for quick conversions and debugging.



   Both interfaces interact with the same backend core - consisting of ex- tractors, converters, and validators - ensuring consistent output across envi- ronments. Additionally, the system employs caching to optimize performance and minimize redundant computations, making WhyteBox both scalable and developer-friendly.

1.3.4 Libraries Used

   The WhyteBox backend leverages a set of robust and reliable Python libraries to enable model extraction, conversion, validation, and API interaction. Each library is carefully chosen for its stability, community support, and compatibility with modern deep learning frameworks:

� PyTorch / TensorFlow / ONNX: Core libraries for loading and ana- lyzing deep learning models.

� Flask: Lightweight web framework used to build the REST API endpoints for model upload and data retrieval.

� JSON / jsonschema: Used for serialization and schema validation of model architecture data.

� NumPy: Provides numerical operations required for tensor and parameter manipulation.

� Click: Enables the design of a structured and user-friendly Command-Line Interface (CLI).

� Werkzeug / Requests: Facilitates HTTP request handling and testing of REST API calls.

� Pickle / TorchScript / ONNX Runtime: Utilized for model parsing, conversion, and interoperability between frameworks.

   These libraries collectively form the backbone of WhyteBox's backend system, ensuring scalability, efficiency, and interoperability across different AI develop- ment environments.



1.3.5 Software Description

   The WhyteBox backend was developed and tested in a modular software en- vironment optimized for flexibility and collaborative development. The primary tools and platforms used are described below.

1.3.5.1 Visual Studio Code

   Visual Studio Code (VS Code) serves as the primary integrated development environment (IDE) for WhyteBox development. Its features such as Python ex- tensions, integrated terminal, Git support, and debugging tools make it highly efficient for backend development. VS Code also supports virtual environments and dependency management, ensuring smooth integration of Python packages and consistent builds across systems.

1.3.5.2 Web Browser Environment

   A modern web browser environment (such as Google Chrome or Microsoft Edge) is used for testing and interacting with the Flask-based REST API. Using browser tools like Postman or native developer consoles, requests can be sent to the API endpoints to verify model uploads, conversions, and JSON outputs. This environment also provides a preview mechanism for upcoming 3D visualization modules, which will be rendered directly in the browser during Phase 2.

1.3.5.3 GitHub and Git Version Control

   Git is employed as the distributed version control system for managing code revisions and maintaining project history. GitHub serves as the central repository for collaborative development, issue tracking, and version management. Branch- ing strategies and pull requests are used to ensure clean integration of new fea- tures, documentation updates, and bug fixes. This workflow supports team-based development, continuous integration, and reproducibility - essential traits for a scalable open-source backend like WhyteBox.







    Chapter 2 Literature Review

2.1 Generalized CNN Visualization Systems

Toward Generalized Visualization System for Various CNN Models: Module and Layer View
Authors: Y. Kim, J. Chun, S. Son and H. Park
Source: ieeexplore.ieee.org


2.1.1 Summary

   This paper introduces CNNExplorer, a generalized visualization system de- signed to make convolutional neural networks (CNNs) interpretable for beginners. The system employs a two-level view architecture: a high-level module view that abstracts complex models into grouped layers and a detailed layer view that vi- sualizes data transformations within each module. Built using PyTorch, Svelte, and Flask, CNNExplorer integrates models from HuggingFace and allows users to visualize inference using their own images. A summative user study validated the tool's usability among non-computer science graduates.

2.1.2 Observations

� Implements a two-level visualization (module and layer view) for CNN in- terpretability.
� Compatible with multiple architectures, including ResNet and GoogLeNet.
� Provides an intuitive web-based interface for model exploration.



2.1.3 Drawbacks

� Limited visualization depth for complex architectures.
� Lacks integration with 3D or temporal data models.
� User study sample size relatively small for broader generalization.


2.2 Explainability in Black-Box Models

A Survey of Methods for Explaining Black Box Models
Authors: R. Guidotti, A. Monreale, F. Turini, D. Pedreschi and F. Giannotti
Source: ieeexplore.ieee.org


2.2.1 Summary

   This survey comprehensively categorizes methods for explaining black-box models in machine learning. It classifies approaches based on the type of ex- planation, the nature of the model, and the input data. Techniques like decision trees, rule extraction, feature importance, and saliency analysis are systematically reviewed. The paper formalizes interpretability problems, offering a framework for model explanation, outcome explanation, inspection, and transparent model design.

2.2.2 Observations

� Provides a structured taxonomy for existing explainable AI (XAI) methods.
� Clarifies the conceptual differences between interpretability, explainability, and comprehensibility.
� Serves as a foundation for future XAI research classification.

2.2.3 Drawbacks

� No standardized definition for "explanation" or "comprehensibility".
� Limited empirical evaluation of interpretability measures.
� Focused mainly on tabular, image, and text data, ignoring other modalities.



2.3 Lightweight CNN

LCNN: Lightweight CNN Architecture for Software Defect Feature Identification Using Explainable AI
Authors: M. Begum et al.
Source: ieeexplore.ieee.org


2.3.1 Summary

   This paper proposes a lightweight convolutional neural network (LCNN) ar- chitecture for software defect prediction using the NASA CM1 dataset. Both 1D-CNN and 2D-CNN models are tested, with 1D-CNN preferred due to its compatibility with explainable AI (XAI) tools such as LIME and SHAP. The approach identifies root causes of software defects through interpretable visual explanations, providing transparency to model predictions.

2.3.2 Observations

� Incorporates XAI methods (LIME and SHAP) for CNN explainability.
� Achieves high accuracy with computational efficiency.
� Highlights LIME's superiority in visualizing tabular CNN predictions.

2.3.3 Drawbacks

� Dataset dependency limits generalization.
� 2D-CNN excluded due to interpretability constraints.
� Lack of large-scale validation across multiple datasets.


2.4 Explainable AI for Non-Experts

Unraveling ML Models of Emotion With NOVA: Multi-Level Explain- able AI for Non-Experts
Authors: A. Heimerl, K. Weitz, T. Baur and E. Andr�e
Source: ieeexplore.ieee.org



2.4.1 Summary

   This work introduces NOVA, an interactive annotation tool leveraging coop- erative machine learning (CML) and explainable AI (XAI) to support emotion recognition by non-experts. Using semi-supervised learning, NOVA iteratively trains models with partial human-labeled data, enhancing model trust and in- terpretability through LIME-based visualizations and confidence metrics. User studies demonstrate improvements in trust and understanding among non-expert participants.

2.4.2 Observations

� Integrates human-in-the-loop ML with XAI visualization.
� Empowers non-experts to annotate and refine ML models interactively.
� Psychological evaluation validates its usability and trust enhancement.

2.4.3 Drawbacks

� Visual explanations may still lead to subjective interpretations.
� Limited to emotion recognition tasks.
� Requires iterative human involvement, increasing annotation time.


2.5 Gradient-Based Visual Explanations

Grad-CAM: Visual Explanations from Deep Networks via Gradient- Based Localization
Authors: R. R. Selvaraju, M. Cogswell, A. Das, R. Vedantam, D. Parikh and
D. Batra
Source: ieeexplore.ieee.org


2.5.1 Summary

   Grad-CAM generates class-discriminative visual explanations for CNN-based models using the gradients of target concepts flowing into the final convolutional



layer. The resulting heatmaps localize critical image regions responsible for pre- dictions. When combined with guided backpropagation, it forms high-resolution visualizations, enhancing interpretability across classification, captioning, and vi- sual question answering tasks.

2.5.2 Observations

� Applicable to any CNN model without architectural modifications.
� Highlights spatial regions influencing model predictions.
� Improves user understanding of CNN decision mechanisms.

2.5.3 Drawbacks

� Produces coarse heatmaps with limited detail.
� Less effective for early-layer visualization.
� Struggles with images containing multiple instances.


2.6 Interpretable CNN Architectures

Interpretable CNNs for Object Classification
Authors: Q. Zhang, X. Wang, Y. N. Wu, H. Zhou and S.-C. Zhu
Source: ieeexplore.ieee.org


2.6.1 Summary

   This paper proposes an interpretable CNN architecture that enforces each high-level filter to represent a specific object part rather than abstract patterns. A mutual information-based filter loss and masking mechanism ensure semantic consistency, allowing filters to correspond directly to object components without explicit part annotations. This proactive interpretability model contrasts with post-hoc explanation methods.

2.6.2 Observations

� Learns part-level interpretable filters during training.


� Provides intrinsic transparency compared to post-hoc XAI.
� Offers potential for integration with 3D visual representations.

2.6.3 Drawbacks

� Limited to single-object, uncluttered images.
� Reduced robustness against adversarial noise.
� May slightly lower classification accuracy on large-scale datasets.


2.7 Discriminative Localization in CNNs

Learning Deep Features for Discriminative Localization Authors: B. Zhou, A. Khosla, A. Lapedriza, A. Oliva and A. Torralba Source: ieeexplore.ieee.org

2.7.1 Summary

   This paper presents Class Activation Mapping (CAM), showing that CNNs trained for classification can also perform object localization using global aver- age pooling (GAP). By summing weighted feature maps, CAM identifies image regions that contribute to specific class predictions, enabling weakly supervised localization without bounding box annotations.

2.7.2 Observations

� Introduces CAM as a simple yet powerful localization method.
� Demonstrates generalization across visual recognition tasks.
� Efficient-requires only a single forward pass.

2.7.3 Drawbacks

� Requires specific GAP-based architectures.
� Minor drop in classification accuracy compared to baseline models.
� Limited localization resolution in deeper networks.



2.8 Understanding CNNs through Deconvolu- tion
Visualizing and Understanding Convolutional Networks Authors: M. D. Zeiler and R. Fergus
Source: ieeexplore.ieee.org


2.8.1 Summary

   This foundational work introduces the deconvolutional network (deconvnet) to visualize CNN feature activations. By projecting feature maps back to input space, it reveals the input patterns that excite specific neurons at each layer, offering a detailed view of hierarchical feature learning. Occlusion analysis is also performed to identify crucial input regions affecting predictions.

2.8.2 Observations

� Pioneering method for visualizing learned CNN features.
� Helps identify important receptive fields and debugging architecture issues.
� Demonstrates hierarchical feature learning from edges to objects.

2.8.3 Drawbacks

� Visual reconstructions are approximate and not generative.

� Computationally intensive for deep networks.

� Limited insight into temporal or 3D CNNs.







Chapter 3
Design and Implementation

   The WhyteBox backend is designed to extract, convert, and validate neural network models from multiple deep learning frameworks and prepare them for interactive visualization and analysis. The system ensures modularity, scalability, and real-time responsiveness, enabling users to upload models, convert them to a universal JSON format, validate the structural integrity of the data, and interact with it through both command-line and REST API interfaces. The architecture is optimized for flexibility and future expansion, supporting planned integration with 3D visualization modules and Explainable AI (XAI) techniques to provide deeper insights into model behavior.


3.1 System Architecture

   The overall architecture of WhyteBox consists of four primary modules: Ex- tractors, Converters, Validators, and Interfaces, each performing a distinct role in the processing pipeline. The Extractor module is responsible for identifying the framework of the input model, which could be PyTorch, Keras, or ONNX, and retrieving essential architectural information including layer types, input and output shapes, trainable parameters, connectivity between layers, and model metadata such as name, version, and size. Once the model is extracted, the Converter module transforms the data into a universal JSON format that is in- dependent of the originating framework. This module also enriches the data with visualization metadata, including layer positions, colors, and groupings, while preserving sequential and skip connections. The Validator module ensures that



the JSON output conforms to a predefined schema, checking for missing fields, incompatible data types, and broken connections to guarantee correctness be- fore output delivery. Users can interact with the system via two interfaces: a REST API that handles online uploads, model conversion, and JSON retrieval, and a command-line interface that enables local model processing with detailed logging, selective layer conversion, and real-time feedback. Complementing these modules, the caching system stores processed models and JSON outputs, reducing redundant computations and improving efficiency for repeated uploads.


Figure 3.1: System Architecture of WhyteBox Backend



3.2 System Components

   The Extractor module forms the first stage of the workflow. It automatically identifies the model framework and extracts a detailed representation of the model architecture. This includes recognizing layer types such as convolutional, linear, or pooling layers, determining input and output tensor shapes, counting trainable



parameters, and mapping connectivity and skip connections. The intermediate representation generated by the Extractor serves as input to the Converter mod- ule, which standardizes the data into a universal JSON schema. The Converter adds visualization-specific attributes that allow each layer to be correctly dis- played in future 3D visualization modules, including color coding, grouping, and spatial layout information. Following conversion, the Validator examines the JSON data to ensure that all required fields are present, data types are consis- tent, and layer connections are valid. Only after passing validation is the data made available to the Interfaces, which provide users with interactive access. The REST API supports web-based model uploads, conversion requests, and JSON downloads, whereas the CLI allows local execution, verbose logging, and selective conversion of layers for debugging. The caching system enhances performance by storing previously processed models and outputs, reducing processing time for repeated conversions and supporting versioning of updated models to maintain a history of changes.


3.3 Workflow

   The WhyteBox backend follows a modular and sequential workflow designed to ensure reliability, scalability, and real-time performance. Initially, users upload models through the REST API or provide them via CLI commands. Supported formats include PyTorch's .pt, Keras's .h5, and ONNX's .onnx files. The sys- tem automatically detects the framework of the uploaded model and selects the appropriate Extractor. The Extractor then parses the model, retrieving detailed information about each layer, its parameters, and connectivity. This interme- diate representation is passed to the Converter, which produces a framework- independent JSON file enriched with visualization metadata and structured for 3D rendering. The Validator then checks the JSON against a predefined schema, ensuring structural integrity and correctness. Once validation is complete, the output is either returned via the REST API or saved locally through the CLI. To improve performance, the caching module stores previously processed models, enabling faster retrieval for repeated uploads and reducing redundant computa-



tion.


3.4 Interactive Model Conversion

   The system supports interactive model conversion through both the REST API and CLI interfaces.  Through the REST API, users can upload models, track real-time conversion progress, and request partial or selective layer conver- sion. The CLI provides a local interface for inspecting layer hierarchies, reviewing detailed logs, and performing selective conversions. This interactive capability al- lows users to debug, analyze, and visualize model data efficiently, while ensuring that the JSON output is reliable and conforms to schema specifications. The inte- gration of caching ensures that repeated conversions are faster, and the enriched metadata prepares the system for seamless connection to future 3D visualization and XAI modules.

Figure 3.2: Interactive Model Conversion Flow in WhyteBox



3.5 Caching and Optimization

   The caching and optimization module enhances the system's performance and scalability. By storing previously processed models and their corresponding JSON outputs, the system avoids redundant computation, reducing latency and im- proving throughput for repeated operations. Versioning support allows updated models to be processed efficiently while preserving historical outputs. These fea- tures collectively ensure that WhyteBox can handle large models in real time and provide a strong foundation for Phase 2, which will focus on 3D interactive visualization and advanced analysis.







      Chapter 4 Results and Discussion

4.1 Phase 1: Interactive Model Conversion

   In the first phase of the WhyteBox project, the backend system was developed to extract, convert, and validate neural network models from multiple frameworks, including PyTorch, Keras, and ONNX. The core objective of this stage was to ensure that models of varying architectures and complexities could be seamlessly transformed into a framework-agnostic JSON representation, preserving all essen- tial information such as layer configurations, parameters, inter-layer connectivity, activation functions, and metadata.
   To achieve this, the system integrated model introspection and graph parsing techniques to automatically traverse the computation graph, detect sequential and skip connections, and record input-output tensor shapes for every layer. Robust validation mechanisms were implemented to cross-check model integrity and verify that the converted structure matched the original network topology.
